### 2. 主要思想

从题名可以判断，论文关注的是 **高阶互补云 API 推荐**，并使用 **个性化张量分解** 建模多维交互关系。相比普通矩阵分解，它更适合表达：
- 开发者 / Mashup / API 的多方关系
- 互补推荐中“组合上下文”对结果的影响
- 不同个体对互补 API 的差异化偏好

---

### 3. 我对方法的推断性理解

结合这一研究方向的常见做法，我认为论文很可能是：
- 用张量而不是二维矩阵表示高阶交互结构。
- 在分解过程中加入个性化因子，刻画不同开发者或不同 Mashup 场景下的互补偏好。
- 利用高阶关系提升“API 与 API 如何搭配”的建模能力，而不是只看单 API 匹配。

这一点和传统协同过滤相比，更适合补全复杂服务组合。

---

### 4. 我认为值得关注的细节

互补推荐天然是高阶问题，因为某个 API 的价值往往取决于“它和谁一起被调用”。如果只做 pairwise 相似度，很多组合依赖是学不到的，因此张量分解的建模方向是合理的。
