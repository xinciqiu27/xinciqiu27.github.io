# Extending Process Intelligence with Quantity-Related Process Mining

## 论文笔记

---

## 一、论文概述
论文题目：Extending Process Intelligence with Quantity-Related Process Mining
作者：Nina Graves, Tobias Brockhoff, István Koren, Wil M. P. van der Aalst
期刊/会议：ICPM 2024 Workshops, LNBIP 533, pp. 337-349, 2025
论文链接：https://doi.org/10.1007/978-3-031-82225-4_25
论文代码：https://github.com/rwth-pads/qrpm.git

---

## 二、研究问题与动机
这篇论文虽然不属于 Web API 推荐主线，但它讨论的是一个很典型的建模问题：很多物流流程的控制逻辑，不仅依赖可识别对象状态，还依赖**数量状态**。例如：
- 库存低于阈值时触发补货；
- 首次发货未满足全部订单时触发二次发货。

传统对象中心过程挖掘（OCPM）默认事件执行主要依赖对象及其属性，而对“不可唯一标识实体的数量变化”支持不足。QRPM 就是在这个缺口上提出扩展框架。

---

## 三、方法设计
### 1. 核心扩展对象
论文提出 Quantity-Related Process Mining（QRPM），扩展了两类基础对象：
- `object-centric event logs`
- `object-centric Petri nets`

扩展目标是让它们能够显式表达“数量状态”和“数量依赖”。

### 2. 为什么数量状态重要
作者以库存管理流程（IMP）为例说明：补货是否触发、第二次发货是否发生，并不由单个对象生命周期决定，而由特定物品数量决定。因此，若只跟踪对象，不跟踪数量，就无法完整还原流程控制逻辑。

### 3. 理论定义
论文正式引入：
- `counter`：表示不同 item type 的数量；
- `collection point`：表示仓库、计划系统等数量收集点；
- `collection counter`：记录不同 collection point 上的数量状态。

在此基础上，作者总结了三类 quantity-related 过程属性：
- 数量变化；
- 执行的数量依赖性；
- 操作的数量依赖性。

### 4. 工具实现
论文不仅提出概念，还实现了 Python 工具 `QRPM`，可以：
- 从 QEL 中发现 q-net；
- 检测事件的数量状态；
- 可视化数量依赖行为。

这让论文不只是形式化定义，而是具备原型可用性。

---

## 四、实验设计与结果
### 1. 验证方式
论文更偏方法论和框架验证，而不是常规 benchmark 排名。它通过一个公共物流流程示例展示：
- 扩展后的事件日志类是否能表达数量状态；
- q-net 是否能反映数量依赖；
- QRPM 工具是否能在实践中工作。

### 2. 主要结论
论文结论是：数量状态可以从扩展日志中被检测出来，且扩展后的对象中心流程模型可以表达传统 OCPM 很难呈现的数量依赖逻辑。

### 3. 我认为最有价值的点
QRPM 最重要的地方在于它说明：如果关键控制变量没有进入建模对象，再强的过程分析算法也只能看到一半流程真相。

---

## 五、论文亮点
- 对 OCPM 的扩展动机非常清晰，和物流场景强相关。
- 形式化定义比较完整，不只是口头提出“数量很重要”。
- 有原型系统支撑，具备实际可验证性。
- 为后续资源约束型流程分析提供了一个结构化起点。

---

## 六、局限性与可改进点
- 论文偏方法框架验证，缺少大规模 benchmark 量化对比。
- 主要场景集中在物流与库存流程，向其他领域迁移仍需进一步验证。
- 数量状态的自动抽取在复杂业务系统中可能成本较高。
- 与传统推荐系统的关联较弱，更适合作为方法论背景文献。

---

## 七、我的收获
QRPM 对我的提醒非常有价值：很多任务效果不好，不是模型能力不足，而是关键状态根本没被显式建模进去。对服务推荐来说，这种“状态缺失”同样可能发生在资源约束、调用配额、上下文数量等变量上。

---

## 八、继续追问的问题
1. 数量状态能否迁移到服务编排或 API 调度任务中？
2. 当数量变量很多且相互耦合时，QEL 和 q-net 会不会过于复杂？
3. QRPM 能否与异常检测结合，发现“数量逻辑异常”的流程片段？
4. 这种数量依赖建模能否启发带资源约束的推荐系统设计？
