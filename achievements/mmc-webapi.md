# Multi-Modal Complementary Web API Recommendation with Attentive Context Modeling

## 基本信息

- 论文题目：Multi-Modal Complementary Web API Recommendation with Attentive Context Modeling
- 会议：2026 29th International Conference on Computer Supported Cooperative Work in Design, CSCWD 2026
- 类别：CCF C 类会议
- 作者身份：第一作者
- 研究方向：Web API Recommendation, Multi-Modal Recommendation, Complementary Recommendation

---

## 摘要信息

这篇工作面向的是**互补式 Web API 推荐**问题。与传统“根据需求文本找相似 API”的任务不同，这个场景中开发者往往已经选好了一部分 API，但还不知道下一步应该补充哪些 API，才能与现有组合协同完成 Mashup 功能。论文指出，已有方法要么依赖显式需求文本，要么主要从单一文本模态中学习语义信息，因此难以充分理解 Web API 之间的互补关系，也难以建模“不同已选 API 对候选 API 的作用强弱并不相同”这一上下文特征。

为了解决这个问题，论文提出 **MoCoACM**。该方法一方面使用 Sentence-BERT 编码 API 功能描述，再结合 LightGCN 从 API 相似图中学习结构语义，形成统一的多模态表示；另一方面引入候选感知的注意力机制，对已选 API 集合进行动态加权，突出那些与当前候选 API 更具互补性的上下文信息；最后再用 SANFM 建模低阶和高阶特征交互，预测候选 API 的互补概率。实验结果表明，MoCoACM 在真实数据集上取得了 **MRR = 0.3064**、**Recall@10 = 0.5636** 的表现，相比当时最强方法 **CoW AR-SANFM** 分别提升 **2.78%** 和 **5.01%**，说明多模态表示学习与上下文注意力建模对于互补推荐是有效的。

## 研究背景

在真实的软件开发过程中，开发者通常需要组合多个功能各异的 Web API 来完成复杂业务逻辑，因此推荐系统不仅要回答“哪个 API 和当前需求相似”，还要回答“哪个 API 能与已经选中的 API 形成有效协同”。

论文特别强调，现有 Web API 推荐研究大多针对**需求驱动**场景：开发者先给出较明确的功能需求，然后系统再去匹配合适的 API。这类方法虽然在功能相似性建模上已经较成熟，但并不完全适用于互补推荐场景，因为互补推荐更关注 API 之间能否组合、能否协作，而不是它们在文本描述上是否相似。

---

## 核心问题

该工作关注的问题是：

给定开发者已经选择的一组 Web API，如何推荐新的、具有互补价值的 Web API？

这不同于传统的单一 API 推荐，因为模型需要理解：

- 已选 API 集合所隐含的功能上下文，而不是仅依赖显式需求描述；
- 候选 API 与不同已选 API 之间的互补强弱，而不是简单平均看待整个上下文；
- API 文本语义和结构关系如何共同支持互补性判断；
- 在长尾 API 较多、组合关系复杂的情况下，如何仍然学到稳定的推荐表示。

---

## 核心方法

论文中的方法框架主要包含四个部分：

1. **基于组合关系的数据标注**  
   利用真实 Mashup 中 Web API 的共现关系自动构造训练样本：同一 Mashup 中共同出现的 API 组合被视为互补关系，未共同出现且不满足组合条件的 API 被构造为负样本。

2. **多模态服务表示学习**  
   使用 Sentence-BERT 对 API 的功能描述进行编码，获得文本语义表示；再根据 API 之间的文本相似度构建 API-API 相似图，并用 LightGCN 在图上聚合邻居信息，学习结构语义表示；最后将两者拼接成统一的多模态向量。

3. **注意力式上下文建模**  
   对于一个候选 API，模型不会把所有已选 API 等权处理，而是通过注意力机制动态学习“哪些已选 API 对当前候选 API 更重要”，从而生成候选感知的上下文表示。

4. **互补推荐建模**  
   将候选 API 表示与上下文表示一起输入 SANFM，显式建模低阶与高阶特征交互，最终输出互补概率。

我理解这篇论文真正解决的关键，不只是“把文本和图拼起来”，而是让模型同时看到：**API 本身的语义、API 之间的结构关联，以及候选 API 在特定上下文中的互补位置**。

## 实验结果

论文在真实数据集上的实验结果显示，MoCoACM 相比当时的最强基线 **CoW AR-SANFM** 在多个指标上都有提升，其中最核心的结果是：

- MRR：从 0.2981 提升到 **0.3064**
- Recall@10：从 0.5367 提升到 **0.5636**
- 相对提升：**MRR +2.78%**、**Recall@10 +5.01%**

论文还做了消融实验，分别比较了 **仅文本表示**、**仅结构表示** 和 **文本+结构的多模态融合表示**。结果表明，多模态融合在各项指标上都优于单一模态，这说明：

- 文本语义有助于捕捉 API 功能层面的细粒度差异；
- 图结构信息有助于挖掘 API 之间的协同关系和全局关联；
- 注意力机制能够更准确地建模“哪个已选 API 对当前候选 API 更关键”。
